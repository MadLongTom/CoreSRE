# Feature Specification: 单 Agent 对话界面

**Feature Branch**: `007-agent-chat-ui`  
**Created**: 2026-02-10  
**Status**: Draft  
**Input**: User description: "我需要有一个单Agent对话的界面，可以选择Agent，选择对话记录（注意一个对话记录开始之后就不能再切换Agent）"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 发起新对话 (Priority: P1)

用户进入对话页面，从已注册的 Agent 列表中选择一个 Agent，然后发送第一条消息以开始一轮新的对话。系统将消息发送给选定的 Agent，并将 Agent 的回复实时展示在对话界面中。对话开始后，Agent 选择器被锁定，不可更改。

**Why this priority**: 这是整个对话功能的核心路径。没有"发起新对话"的能力，其他所有功能都无法使用。

**Independent Test**: 可以通过选择一个 Agent、输入消息并发送来独立测试。发送后能看到 Agent 回复即验证成功。

**Acceptance Scenarios**:

1. **Given** 用户已进入对话页面且系统中存在至少一个状态为"在线"的 Agent, **When** 用户从下拉选择器中选择一个 Agent, **Then** 系统显示该 Agent 的名称和类型，消息输入框变为可用状态
2. **Given** 用户已选择一个 Agent 且输入了非空消息, **When** 用户点击发送按钮或按 Enter, **Then** 用户消息显示在对话区域，系统将消息发送给 Agent 并展示 Agent 的回复
3. **Given** 用户已发送第一条消息（对话已开始）, **When** 用户尝试更改 Agent 选择器, **Then** Agent 选择器处于禁用/锁定状态，无法切换
4. **Given** 用户尚未选择任何 Agent, **When** 用户尝试在输入框中输入消息, **Then** 输入框为禁用状态，提示用户先选择 Agent

---

### User Story 2 - 继续历史对话 (Priority: P1)

用户可以查看自己的对话记录列表，选择一条历史对话后恢复到当时的上下文继续对话。历史对话与特定 Agent 绑定，打开后 Agent 选择器显示对应的 Agent 并处于锁定状态。

**Why this priority**: 对话的持久化和恢复能力与发起新对话同等重要，否则每次进入页面都要重新开始，严重影响用户体验。

**Independent Test**: 在已有至少一条历史对话的前提下，从对话列表中点击一条记录，验证历史消息正确展示、Agent 正确绑定、且可以继续发送消息。

**Acceptance Scenarios**:

1. **Given** 用户已进入对话页面, **When** 页面加载完成, **Then** 侧边或顶部显示对话记录列表，按最近活跃时间倒序排列
2. **Given** 对话记录列表中存在历史对话, **When** 用户点击一条历史对话, **Then** 对话区域展示该对话的完整历史消息，Agent 选择器显示对应 Agent 名称并处于锁定状态
3. **Given** 用户已打开一条历史对话, **When** 用户输入新消息并发送, **Then** 消息被追加到该对话中，Agent 基于完整上下文进行回复
4. **Given** 用户正在一条对话中, **When** 用户点击对话列表中的另一条对话, **Then** 当前对话自动保存，界面切换到新选择的对话

---

### User Story 3 - 创建全新对话 (Priority: P2)

用户可以随时点击"新建对话"按钮，清空当前对话区域，重新选择 Agent 并开始一段全新的对话。

**Why this priority**: 有了发起和继续对话的核心能力后，用户需要明确的入口来开始新的对话，而不是刷新页面。

**Independent Test**: 在正在进行对话时，点击"新建对话"按钮，验证对话区域被清空、Agent 选择器恢复为可选状态。

**Acceptance Scenarios**:

1. **Given** 用户正在与某个 Agent 对话, **When** 用户点击"新建对话"按钮, **Then** 对话区域清空，Agent 选择器恢复为可选状态，输入框保持可用
2. **Given** 用户点击了"新建对话", **When** 用户选择一个新的 Agent 并发送消息, **Then** 系统创建一条新的对话记录，旧对话保留在对话列表中

---

### User Story 4 - 对话消息的流式展示 (Priority: P2)

Agent 的回复以流式方式逐步展示在对话界面中，用户不必等待完整回复生成完毕即可开始阅读。

**Why this priority**: 流式展示是现代 AI 对话界面的标准体验，显著提升用户感知响应速度。但功能上非阻塞性需求，可以在基本对话完成后迭代。

**Independent Test**: 发送一条消息后，观察 Agent 回复是否逐字/逐块出现而非一次性全部展示。

**Acceptance Scenarios**:

1. **Given** 用户已发送消息, **When** Agent 开始生成回复, **Then** 对话区域立即显示一个加载指示，回复内容逐步出现
2. **Given** Agent 正在流式输出回复, **When** 输出完成, **Then** 消息气泡完成渲染，加载指示消失

---

### User Story 5 - 删除对话记录 (Priority: P3)

用户可以删除不再需要的对话记录。删除前需要确认以防误操作。

**Why this priority**: 属于辅助管理功能，不影响核心对话体验。

**Independent Test**: 在对话列表中，对一条对话执行删除操作，验证确认弹窗出现、确认后对话从列表中消失。

**Acceptance Scenarios**:

1. **Given** 对话列表中存在对话记录, **When** 用户点击某条对话的删除按钮, **Then** 系统弹出确认对话框，提示"确认删除此对话？"
2. **Given** 确认对话框已显示, **When** 用户点击确认, **Then** 对话记录被删除，列表刷新，若删除的是当前对话则界面切换到"新建对话"状态
3. **Given** 确认对话框已显示, **When** 用户点击取消, **Then** 对话框关闭，对话记录保留

---

### Edge Cases

- 当选定的 Agent 离线或不可用时，发送消息后应显示错误提示而不是无限等待
- 当 Agent 在对话中途被删除时，历史对话仍可查看但不能继续发送新消息，提示"Agent 已不可用"
- 当用户快速连续发送多条消息时，系统应按顺序处理，防止消息乱序
- 当网络断开时，发送失败的消息应显示重试按钮
- 当对话历史非常长（超过 100 条消息）时，应支持滚动加载而不是一次加载全部
- 当没有任何已注册 Agent 时，对话页面应显示友好提示引导用户先注册 Agent
- 当用户发送空消息或纯空白消息时，发送按钮应保持禁用状态

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须提供一个专用的对话页面，包含 Agent 选择器、对话记录列表和消息对话区域
- **FR-002**: Agent 选择器必须展示所有已注册的 Agent，显示名称和类型
- **FR-003**: 用户发送第一条消息后，Agent 选择器必须锁定，在当前对话中不可更改
- **FR-004**: 系统必须支持向选定 Agent 发送文本消息并展示 Agent 的回复
- **FR-005**: 系统必须持久化每一轮对话，包括对话关联的 Agent、所有消息内容和时间戳
- **FR-006**: 系统必须提供对话记录列表，按最近活跃时间倒序排列，展示对话摘要信息（Agent 名称、最后一条消息预览、时间）
- **FR-007**: 用户选择历史对话时，系统必须恢复完整的消息历史和 Agent 上下文
- **FR-008**: 系统必须提供"新建对话"功能，允许用户清空当前对话并重新选择 Agent
- **FR-009**: 系统必须支持删除对话记录，删除前需确认
- **FR-010**: Agent 回复过程中必须展示加载指示，告知用户正在等待回复
- **FR-011**: 消息发送失败时，系统必须显示错误提示并提供重试选项
- **FR-012**: 对话页面必须可通过侧边导航栏进入

### Key Entities

- **对话（Conversation）**: 代表用户与某个 Agent 之间的一轮完整对话。包含唯一标识、关联的 Agent、创建时间、最后活跃时间。一旦创建，关联的 Agent 不可更改。
- **消息（Message）**: 对话中的单条消息。包含消息内容、发送方（用户或 Agent）、时间戳。按时间顺序排列构成对话历史。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户从进入对话页面到发送第一条消息的操作时间不超过 15 秒
- **SC-002**: Agent 回复的首字延迟（从发送到看到第一个字符）不超过 3 秒
- **SC-003**: 对话记录列表加载时间不超过 1 秒
- **SC-004**: 用户可以在 2 步操作内创建新对话（点击新建 → 选择 Agent）
- **SC-005**: 切换历史对话后，消息历史在 2 秒内完整展示
- **SC-006**: 所有对话状态在页面刷新后完整保留，无数据丢失

## Assumptions

- Agent 选择器中展示所有状态的 Agent（包括在线、离线等），但仅在线 Agent 可以进行对话
- 对话历史长度无上限，但前端展示时采用虚拟滚动或分页加载以保证性能
- 每个对话绑定一个 Agent，不支持多 Agent 协同对话（本次需求明确为"单 Agent 对话"）
- 消息格式为纯文本，暂不支持富文本、图片或文件附件
- 系统不区分用户身份（单用户系统），所有对话记录属于当前用户
